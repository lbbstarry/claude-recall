# What I learned indexing 6,500 turns of Claude Code conversation

*April 2026 · `claudegrep` v0.2*

I spent three weeks building a search index over my own Claude Code history.
What I didn't expect: the **eval set** turned out to be the most valuable
artifact in the repo, more than the code.

This is a writeup of what's in `tests/fixtures/queries.jsonl`, what the
numbers look like, and what surprised me.

## The corpus

127 sessions across 6 projects in `~/.claude/projects/`. After per-turn
chunking (one user message + the immediately following assistant
turn), that's **6,593 chunks**, average ~1.2k tokens each.

Per-turn chunking is a deliberate choice. Per-message loses the Q-A pairing
that makes a chunk useful. Sliding-window inflates the index 3-5×. A turn fits
inside bge-m3's 8k context with no truncation, and still has enough text for
BM25 to bite into.

## The eval set

51 hand-labeled `(query, relevant_chunk_id)` pairs, sampled from my own
history with a CLI labeling tool. 22 unique target chunks. Apache-2.0 in
`tests/fixtures/queries.jsonl`.

Median query length: **5 characters**. Examples:

- `prefab`
- `figma示例`
- `zgame标准`
- `rrf`
- `sqlite vec`

This is what dev queries actually look like — vague, short, often a single
project-specific noun, sometimes mixed-language. Any eval that uses long
natural-language questions overstates how well retrieval works in practice.

## The numbers

CPU-only on a WSL2 Ryzen laptop. p95 latencies measured on warm cache.

| Method | Recall@10 | MRR | nDCG@10 | p95 ms |
|---|---:|---:|---:|---:|
| BM25 (FTS5) | 0.216 | 0.125 | 0.148 | 2 |
| Vector (`bge-small-zh-v1.5`, 512d) | 0.353 | 0.175 | 0.217 | 13 |
| Hybrid (RRF, k=60) | 0.392 | 0.175 | 0.228 | 16 |
| **Hybrid + rerank (`bge-reranker-base`)** | **0.471** | **0.230** | **0.289** | 214 |

Reproduce: `python -m claude_recall.eval.run`.

## Five things that surprised me

**1. BM25 alone is genuinely bad on short dev queries.**
21.6% Recall@10 means four out of five short queries don't surface their
target in the top ten lexical hits. Dev conversation has too many
near-duplicate filenames, error fragments, and re-quoted code for term
overlap to be discriminative.

**2. Vector retrieval beats BM25 by 64% on this task.**
The `bge-small-zh` model is small (512 dimensions, ~120MB) and it still
catches paraphrase that FTS5 misses. The win is biggest on Chinese-language
queries, which is unsurprising — the model was trained for that.

**3. RRF hybrid only adds ~4 points over pure vector.**
Less than I expected. On long queries hybrid usually adds 8–10 points. On
short queries the BM25 column has so few hits to contribute that the fusion
benefit collapses.

**4. The reranker is the biggest single jump.**
+8 points Recall@10, +6 nDCG. The cross-encoder sees the full query-document
text (no embedding bottleneck) and reorders aggressively. The cost is real:
214ms p95 vs 16ms without rerank. For interactive search at top-10 this is
fine; for "as-you-type" UX you want the bi-encoder result first and the
rerank result a moment later.

**5. Recall@10 of 0.471 sounds bad until you read the queries.**
For five-character queries with no context, half-correct in the top ten is
already crossing the threshold from useless to useful. I notice I keep
going back to the tool. The number is bad on a textbook benchmark; on this
distribution it's the working floor.

## What didn't help

- **Increasing `limit`** beyond ~30 candidates before reranking.
  The reranker is good enough that giving it more candidates didn't lift
  Recall@10. It pulls the right document out of 30 just as well as out of
  100, and gets cheaper.
- **Stripping code blocks before embedding.**
  Tried. Hurt every metric. Code-symbol queries (`prefab`, `chunks_vec`)
  rely on the code being in the embedded text.
- **Using `bge-base-zh` (768d) instead of `small`.**
  No measurable Recall@10 win on this set, doubled embedding time. Held off
  until v0.3 when the planned move is `bge-m3` (1024d, 8k context, multilingual).

## What's next

- **`bge-m3` + query expansion** — biggest planned single bump, lands in v0.3.
- **Larger eval set.** 51 is the floor for "not pure noise"; 200+ is needed
  before micro-tuning RRF k or candidate counts has any signal.
- **Per-project eval breakdown** — I suspect cross-project semantic transfer
  is hurting numbers more than I think.

## Lessons I'd take to the next retrieval project

1. **Build the labeled eval set on day 2.** The instinct to "get it working
   first" is a trap. Without numbers you don't know if a change helped.
2. **Use real queries, not synthetic ones.** Dev queries are short and
   weird. Anything generated by an LLM on top of your corpus reads more
   like an exam question than a real search.
3. **Report Recall, MRR, *and* nDCG.** Each catches a different failure
   mode. Recall says "is it in there at all," MRR says "is it near the
   top," nDCG penalises the long-tail of partial relevance.
4. **Latency p95, not p50.** Search feels broken at the tail.
5. **Reranker is non-negotiable above ~5k chunks.** It's a shorter route
   to "actually usable" than a better embedder.

Apache-2.0, on GitHub: <https://github.com/lbbstarry/claudegrep>. Install: `uv tool install claudegrep`.
